    Adaptive Faster Polling

    Capture denser drive paths and sharper charge curves by polling every ~20–30s only while the car is actively driving or charging — without leaving the read-only, no-virtual-key, Workers + Supabase architecture.

    Changelog
    2026-06-18 19:54Initial draft — folds in roast findings: session-close hysteresis, connection discipline, DB compare-and-swap as the primary token guarantee, a consecutive-failure circuit breaker, and a cost pre-flight check.

    Context #

    • Drive paths and charge curves are coarse because the poller samples every 2 minutes — turns render polygonal and the charge ramp/taper is under-sampled.
    • True 500ms Fleet Telemetry was separately evaluated and rejected (it needs a paired virtual key, an always-on mTLS server, and 100–1000× the write volume). This is the read-only-friendly middle ground.
    • Hard constraints: stay read-only (no virtual key, no commands) on Cloudflare Workers + Supabase; the platform cron floors at 1-minute granularity.
    • Battery safety: frequently polling an already-online parked car delays sleep → vampire drain. A sleeping car cannot be woken by a read (it returns 408), so the risk is the parked-online case, not the sleeper.

    Approach #

    The 2-minute cron stays the baseline and watchdog: it keeps detecting activity exactly as today. When it sees a vehicle driving or charging, it arms a per-vehicle burst poller that drives a self-re-arming ~20s (driving) / ~30s (charging) loop, running the same snapshot + sessionization step the cron already uses — so the denser samples flow straight through the existing density-agnostic aggregation. The loop re-arms before each fallible poll, debounces session close behind N consecutive inactive reads, and stops on a clean close or a safety cap — handing idle/asleep back to the cron so the car sleeps normally. The one genuinely new failure mode — two execution contexts touching the atomically-rotating refresh token — is contained by the database compare-and-swap on the token row (the only guard that spans isolates), not by best-effort scheduling.

    Tradeoffs & out of scope #

  1. ACCEPT: ≤2-min lag before the burst poller arms → a drive's first minute or two is captured at baseline resolution.
  2. ACCEPT: ~1.5× Fleet-API calls during active windows → modest cost increase, gated to active periods only.
  3. ACCEPT: session-close hysteresis runs a few extra polls past the true end before closing → negligible at 20s.
  4. ACCEPT: one new stateful component (a per-vehicle burst poller) → small added complexity, but Workers-native and ~$0 incremental.
  5. OUT: idle/asleep cadence is unchanged — no faster polling of parked cars (protects sleep).
  6. OUT: charge curves are not the justification — 60s already captures them; the real win is drive-path smoothness.
  7. OUT: true sub-second / 500ms telemetry — separately rejected (needs a virtual key + an always-on mTLS server).

    8. Open questions #

    • What is the actual current Fleet-API monthly spend versus the $10 credit — and if it is already over, should the cadence start slower than 20s?
    • Does the token store, on a losing compare-and-swap, re-read the winner's freshly-rotated refresh token rather than reuse the stale one (must be verified during implementation)?
    • Should a future "sleep mode" back the idle cron down to a cheap state-only check for long-parked cars to offset the burst poller's added cost, accepting it might miss a charge-start?
    • Should the active cadence be a user-facing setting, or stay hardcoded to begin with?
